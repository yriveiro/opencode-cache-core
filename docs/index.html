Package overview

This package is for OpenCode plugins that keep a local cache of external projects as Git checkouts, optionally build derived artifacts from those checkouts, and search that cache locally through generated tools.

What ships today

Cache spec

A downstream plugin defines sources, optional artifacts, and named search sections in one validated spec.

Generated tools

The package wires update, status, and search tools from that spec and exposes them as an OpenCode plugin.

Persisted state

The runtime writes cache-state.json and search-index.json into the resolved cache directory.

Runtime context

Downstream plugins get helpers for state, index refresh, path resolution, notifications, and subprocess execution.

What a plugin author imports

src/index.ts is the package entrypoint. A downstream plugin usually imports defineGitCacheSpec() and createGitCachePlugin(), then optionally uses lower-level helpers for custom status lines or package-specific tools.

The runtime implementation lives in src/git-cache-plugin.ts, while the rest of the package is split into spec, state, path, search, and notification helpers.

  • Use defineGitCacheSpec() to validate the cache contract.
  • Use createGitCachePlugin() to publish the generated OpenCode plugin.
  • Use the runtime context from extraTools(runtime) when a plugin needs build steps or package-specific commands.

What the spec describes

The cache spec describes everything the runtime needs: where to clone repositories, where artifacts live, which directories are indexed for search, and what names/descriptions the generated tools should use.

The schema also verifies internal references so artifacts and sections always point at real source or artifact IDs.

  spec: GitCacheSpec<TScope>,
): GitCacheSpec<TScope> {

How sections become a search index

buildGitCacheSearchIndex() resolves each section to a concrete base directory. buildIndex() scans that directory with the section's patterns and stores the matched relative file paths.

Each section becomes one search scope in the generated search tool.

Example

A plugin can expose one section for markdown docs in the repository and another section for generated HTML in an artifact directory.

src/indexing.ts Excerpts 
export async function buildIndex<TScope extends string>(input: {
  cacheDir: string;
  sections: Record<TScope, SectionDefinition>;
}): Promise<Index<TScope>> {
  const sections = {} as Record<TScope, Section>;
  const entries = Object.entries(input.sections) as Array<[TScope, SectionDefinition]>;

  for (const [scope, definition] of entries) {
    sections[scope] = {
      baseDir: definition.baseDir,
      files: await scanRelativeFiles(definition.baseDir, definition.patterns),
    };
  }

  return {
    createdAt: new Date().toISOString(),
    cacheDir: input.cacheDir,
    sections,
  };
}

What the runtime context gives you

createRuntimeContext() gives the generated tools and downstream extensions a single place to read state, refresh the index, resolve source and artifact paths, run commands, and send notifications.

This is the object passed into extendStatus(runtime) and extraTools(runtime).

src/git-cache-plugin.ts Excerpts 
function createRuntimeContext<TScope extends string>(input: {...}): GitCacheRuntimeContext<TScope> {
  const indexFile = getGitCacheIndexPath(input.cacheDir);
  const stateFile = getGitCacheStateFile(input.cacheDir);
  const scopes = getGitCacheScopes(input.spec);
  const readySourceId = resolveReadySourceId(input.spec);

  const readState = async (): Promise<GitCacheState> => {
    return loadGitCacheState(input.spec, input.cacheDir, {...});
  };

  const refreshIndex = async (): Promise<Index<TScope>> => {
    const index = await buildGitCacheSearchIndex(input.spec, input.cacheDir);
    await writeIndex(indexFile, index);
    await updateState((state) => {
      state.indexedAt = index.createdAt;
      state.indexFile = indexFile;
    });
    return index;
  };

  const loadIndex = async (): Promise<Index<TScope> | null> => {
    if (!(await pathExists(join(getGitSourceDir(input.spec, input.cacheDir, readySourceId), ".git")))) {
      return null;
    }

    const storedIndex = await readIndex<Index<TScope>>(indexFile);
    return isIndexForScopes(storedIndex, scopes) ? storedIndex : refreshIndex();
  };

  return { ... };
}

Defining a cache

A downstream plugin describes its cache in one Git cache spec. The spec names the tools, defines the repositories to clone, defines optional artifacts built from those repositories, and defines the searchable sections exposed by the search tool.

Cache directory and readiness

  • process.env[spec.envVar] overrides the cache directory when present.
  • Otherwise the cache lives under ~/.cache/opencode/skills/<spec.defaultCacheSubdir>.
  • readySource, or the first ready-marked source, or the first source, determines whether the cache is initialized for status/search.
  • cache-state.json and search-index.json are derived from that resolved cache directory.

Important spec fields

FieldMeaningExample
envVarEnvironment variable that overrides the cache directory.OPENCODE_PYDANTIC_CACHE_DIR
defaultCacheSubdirDefault directory name under ~/.cache/opencode/skills.pydantic
sourcesRepositories cloned into the cache directory.[{ id, url, branch, sparse? }]
artifactsDirectories inside a source tree that contain generated output.[{ id, source, path, readiness? }]
sectionsSearch scopes built from either a source tree or an artifact tree.{ docs: { root, patterns } }
readySourceSource used to determine whether status/search can treat the cache as initialized.repo
updateTool, statusTool, searchToolNames and descriptions for the generated OpenCode tools.{ name, description, ... }

Sections define search scopes

The sections map is the source of truth for search scopes. Each key becomes a selectable scope in the generated search tool.

Each section points to either a source tree or an artifact tree, which lets one cache expose repository text, library source, and built documentation as separate searchable corpora.

Example section map TypeScript 
sections: {
  docs: {
    label: "Docs",
    root: { kind: "source", id: "repo" },
    patterns: ["docs/**/*.md", "docs/**/*.mdx"],
  },
  built: {
    label: "Built",
    root: { kind: "artifact", id: "site" },
    patterns: ["**/*.html", "**/*.json"],
  },
}

Resolved paths in code

resolveGitCacheDir(), getGitSourceDir(), getGitArtifactDir(), and getGitSectionBaseDir() turn the logical spec into concrete on-disk paths.

These helpers are useful both inside the runtime and in downstream package-specific tools.

src/git-cache-plugin.ts and src/git-cache-schema.ts Excerpts 
function resolveGitCacheDir(
  envVar: string,
  defaultCacheSubdir: string,
  homeDirectory = homedir(),
): string {
  const configuredCacheDir = process.env[envVar];
  if (configuredCacheDir != null && configuredCacheDir.length > 0) {
    return configuredCacheDir;
  }

  return join(homeDirectory, ".cache", "opencode", "skills", defaultCacheSubdir);
}

export function resolveReadySourceId<TScope extends string>(
  spec: GitCacheSpec<TScope>,
): string {
  if (spec.readySource != null) {
    return spec.readySource;
  }

  const markedReadySource = spec.sources.find((source) => source.ready === true);
  if (markedReadySource != null) {
    return markedReadySource.id;
  }

  return spec.sources[0]!.id;
}

Generated tools

The generated plugin exposes three tools from the spec plus a narrow cache-directory permission hook. Tool names and descriptions come directly from the downstream plugin's spec.

Tool summary

Update tool

Runs Git sync for every source, refreshes the search index, persists state, and skips work when the cache is still fresh unless force is set.

Status tool

Reports cache directory, state and index files, freshness, source readiness, artifact readiness, and current search corpus counts.

Search tool

Loads the persisted index on demand, supports scope filtering, regex mode, case sensitivity, and returns formatted hits with scope/file/line excerpts.

Permission hook

Allows cache-directory access requests so the generated tools can read and update the local cache.

What the update tool does

    Runtime->>Git: clone / fetch / reset / sparse-checkout
    Runtime->>State: write source state
    Tool->>Index: refreshIndex()
    Tool->>Session: send notification
    Tool-->>Host: return output

Permission matching for cache access

The permission hook only checks external_directory requests whose title or pattern contains the resolved cache directory.

That keeps the allowance precise and predictable even when multiple tools or plugins are active in the same host session.

src/permissions.ts Excerpts 
function matchesPermission(
  cacheDir: string,
  request: PermissionRequest,
): boolean {
  if (request.type !== "external_directory") {
    return false;
  }

  const patterns = Array.isArray(request.pattern)
    ? request.pattern
    : request.pattern
      ? [request.pattern]
      : [];

  return (
    request.title.includes(cacheDir) ||
    patterns.some((pattern) => pattern.includes(cacheDir))
  );
}

What the search tool returns

The generated search tool accepts a query and optional scope/filter flags, loads the persisted index on demand, and formats each hit as [scope] file:line: excerpt.

The lower-level helper still returns structured search results, so downstream code can use the same search engine without the formatted tool output.

src/tools.ts Excerpts 
export function createSearchTool<TIndex, TScope extends string>(input: {
  description: string;
  notificationTitle: string;
  missingOutput: string;
  scopeValues: readonly [typeof ALL_SCOPE, ...TScope[]];
  scopeDescription?: string;
  failureLabel?: string;
  loadIndex: () => Promise<TIndex | null>;
  search: (input: {
    index: TIndex;
    query: string;
    scope?: TScope | typeof ALL_SCOPE;
    regex?: boolean;
    caseSensitive?: boolean;
    limit?: number;
  }) => Promise<SearchResult<TScope>>;
  sendNotification: (sessionID: string, message: string) => Promise<void>;
}): ToolDefinition {
  return tool({
    description: input.description,
    args: {...},
    async execute(args, context: ToolContext) {
      try {
        const index = await input.loadIndex();
        if (index == null) {
          ...
          return input.missingOutput;
        }

        const output = formatSearchOutput(
          args.query,
          await input.search({...}),
        );

        await input.sendNotification(
          context.sessionID,
          buildNotification(input.notificationTitle, output),
        );
        return output;
      } catch (error: unknown) {
        const message = input.failureLabel == null
          ? formatErrorMessage("Invalid search query", error)
          : formatErrorMessage(input.failureLabel, error);
        ...
        return message;
      }
    },
  });
}

How notifications are sent

buildNotification() builds a prompt-style message shared by all generated tools, and createNotificationSender() pushes that message into the active session with noReply enabled.

If notification delivery fails, the runtime falls back to debug logging instead of breaking the tool result itself.

src/notifications.ts Excerpts 
export function buildNotification(title: string, output: string): string {
  return [`▣ ${title}`, "", output].join("\n");
}

export function createNotificationSender(input: {
  client: PluginInput["client"];
  service: string;
}): (sessionID: string, message: string) => Promise<void> {
  return async (sessionID: string, message: string): Promise<void> => {
    try {
      await input.client.session.prompt({
        path: { id: sessionID },
        body: {
          noReply: true,
          parts: [{ type: "text", text: message, ignored: true }],
        },
      });
    } catch (error: unknown) {
      ...
    }
  };
}

Step 3: sync Git sources and persist state

syncGitSource() handles the actual repository update cycle: clone when missing, fetch/reset when present, and optionally manage sparse checkouts.

syncSources() applies that logic to every source in the spec and records revision and readiness details in the persisted state file.

src/git-cache-plugin.ts Excerpts 
async function syncGitSource(
  sourceDir: string,
  source: GitCacheSourceSpec,
): Promise<string[]> {
  const sourceName = getGitSourceDirectoryName(source);
  const installed = await pathExists(join(sourceDir, ".git"));

  if (!installed) {
    if (source.sparse != null) {
      await runCommand("git", ["clone", "--filter=blob:none", "--no-checkout", "--depth", "1", "--branch", source.branch, source.url, sourceDir]);
      await runCommand("git", ["sparse-checkout", "init", "--cone"], sourceDir);
      await runCommand("git", ["sparse-checkout", "set", ...getSparseEntries(source)], sourceDir);
      await runCommand("git", ["checkout", source.branch], sourceDir);
    } else {
      await runCommand("git", ["clone", "--depth", "1", "--branch", source.branch, source.url, sourceDir]);
    }
    return [`  Cloned ${sourceName}`];
  }

  await runCommand("git", ["fetch", "--depth", "1", "origin", source.branch], sourceDir);
  await runCommand("git", ["reset", "--hard", `origin/${source.branch}`], sourceDir);
  return [`  Updated ${sourceName}`];
}

Step 4: assemble the generated tools

createGitCachePlugin() resolves the spec, builds the runtime, and uses the shared tool helpers to publish the update, status, and search tools under the names defined by the downstream plugin's spec.

Any package-specific build or inspection tools can be added through extraTools(runtime) without reimplementing the core lifecycle.

src/git-cache-plugin.ts Excerpts 
export function createGitCachePlugin<TScope extends string>(
  input: GitCachePluginOptions<TScope>,
): Plugin {
  return async ({ client }) => {
    const spec = defineGitCacheSpec(input.spec);
    const cacheDir = resolveGitCacheDir(spec.envVar, spec.defaultCacheSubdir);
    const sendNotification = createNotificationSender({ client, service: spec.service });
    const runtime = createRuntimeContext({
      client,
      spec,
      cacheDir,
      sendNotification,
      maxAgeSeconds: input.maxAgeSeconds ?? DEFAULT_GIT_CACHE_MAX_AGE_SECONDS,
    });

    const tools: Hooks["tool"] = {
      [spec.updateTool.name]: tool({...}),
      [spec.statusTool.name]: createStatusTool({...}),
      [spec.searchTool.name]: createSearchTool<Index<TScope>, TScope>({...}),
    };

    if (input.extraTools != null) {
      Object.assign(tools, input.extraTools(runtime));
    }

    return {
      tool: tools,
      "permission.ask": createPermissionHandler(runtime.cacheDir),
    };
  };
}

Step 5: run the search algorithm

searchIndex() loops through the indexed files for the selected scopes, reads each file, scans line by line, and returns structured hits containing scope, file, line, and excerpt.

The tool wrapper then handles presentation, notifications, and package-specific error labeling around that core search loop.

export async function searchIndex<TScope extends string>(
  index: Index<TScope>,
  query: string,
  options?: SearchOptions<TScope>,
): Promise<SearchResult<TScope>> {
  const scope = options?.scope ?? ALL_SCOPE;

    const section = index.sections[selectedScope];
