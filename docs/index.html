Plugin overview

The plugin wraps cache indexing and search into a small OpenCode-compatible package with explicit configuration resolution, runtime notifications, and a compatibility permission hook.

What ships today

cache_status

Reports readiness, freshness, configured scopes, and indexed file counts when the index can be loaded.

cache_search

Searches the cache index by substring or regular expression, supports scope filtering, and returns excerpts with file and line numbers.

permission.ask

Auto-allows directory access requests when the host is asking about the configured cache directory through the compatibility hook.

How the package is assembled

The default export resolves configuration, builds notification helpers, loads or creates the cache index, then exposes two tools plus the compatibility hook permission.ask.

src/index.ts is the public entrypoint that re-exports the package surface, while the actual plugin assembly lives in src/plugin.ts.

  • The default export resolves configuration from plugin options, environment aliases, and built-in defaults before any tool is created.
  • Index loading stays lazy through createIndexLoader(), so tools can reuse a persisted JSON index when it already exists.
  • The assembled hooks expose two tools plus the narrow permission.ask compatibility handler.

Architecture and feature map

The repository is intentionally split into small files so each stage of cache handling stays independently understandable.

Module responsibilities

Index construction stays scope-aware

buildIndex() receives the resolved cache directory and a scope map. It iterates the configured sections, scans each baseDir with its own glob-style patterns, and stores the resulting relative file list under that scope key.

That structure matters later because search can either scan every section or limit execution to one named scope without recomputing filesystem state.

Feature tie-in

The docs, code, or any other cached corpus can be separated into dedicated scopes with their own base directory and glob patterns.

Lifecycle keeps runtime startup cheap

loadIndex() first attempts to read the persisted JSON index. Only when that file is missing and the ready marker exists does it call the supplied index builder and optionally persist the fresh result.

createIndexLoader() packages that behavior into the lazy loader used by the tools, which keeps startup cheap while still recovering from a missing serialized index.

Configuration and env mapping

The plugin reads from context config first, then environment variables, and finally falls back to safe defaults rooted in the user's cache directory.

Resolution order

  • Nested context.config.cache values win first.
  • Top-level context.config values are read next for compatibility.
  • Environment variables fill in any missing values.
  • Defaults are used when neither config nor env values are present.

Compatibility note

Environment lookup accepts both OPENCODE_CACHE_* names and shorter OC_CACHE_* aliases.

Sections define search scopes

The sections object maps a scope name to a baseDir and a list of file patterns. Each key becomes a selectable search scope in the tool layer.

Use separate sections when you want a clean distinction between docs, source, generated text, or any other cached corpus.

{
  "docs": {
    "baseDir": "/absolute/path/to/cache/docs",
    "patterns": ["**/*.md", "**/*.txt"]
  },
  "code": {
    "baseDir": "/absolute/path/to/cache/code",
    "patterns": ["**/*.ts", "**/*.tsx", "**/*.js"]

Tool behavior and compatibility hooks

The host-facing surface area is small: two tools and one compatibility hook. The implementation focuses on predictable output and lightweight host integration.

Behavior summary

cache_status

Builds a readiness report from the ready marker path, freshness timestamp, index loader, and configured scopes.

cache_search

Loads the index on demand, handles missing-index messaging, runs substring or regex search, then pushes a notification back to the active session.

Notifications

Both tools send a prompt-style message so results show up in the session as well as in the returned tool output.

permission.ask

Only sets output.status = "allow" when the request targets the configured cache directory through the host's permission request payload.

Permission matching is intentionally narrow

The compatibility hook does not blanket-allow all requests. It first checks that the host is asking for external directory access and then tests whether the request title or patterns include the configured cache directory.

That makes the behavior predictable for hosts that still use permission.ask without affecting unrelated requests.

Step 4: run the search algorithm

searchIndex() loops through each selected scope, reads each indexed file, splits content into lines, and tests the query against each line.

Hits carry scope, file, line, and excerpt data so the tool wrapper can present useful output without re-reading the file again.

 32   const caseSensitive = options?.caseSensitive ?? false;
 33   const limit = options?.limit ?? 20;
 34   const matcher = buildMatcher(query, regex, caseSensitive);
 35   const scopes = scope === ALL_SCOPE ? (Object.keys(index.sections) as TScope[]) : [scope];
 36   const hits: SearchHit<TScope>[] = [];
 37   let scannedFiles = 0;
 38 
 39   for (const selectedScope of scopes) {
 40     const section = index.sections[selectedScope];
 41 
 42     for (const relativeFile of section.files) {
 43       scannedFiles += 1;
 44 
 45       let content: string;
 46 
 47       try {
 48         content = await readFile(join(section.baseDir, relativeFile), "utf8");
 49       } catch {
 50         continue;
 51       }
 52 
 53       const lines = content.split(/\r?\n/u);
 54 
 55       for (const [lineIndex, line] of lines.entries()) {
 56         matcher.lastIndex = 0;
 57         if (!matcher.test(line)) {
 58           continue;
 59         }
 60 
 61         hits.push({
 62           scope: selectedScope,
 63           file: relativeFile,
 64           line: lineIndex + 1,
 65           excerpt: trimExcerpt(line),
 66         });
 67 
 68         if (hits.length >= limit) {
 69           return {
 70             hits,
 71             scannedFiles,
 72             scope,
 73           };
 74         }
 75       }
 76     }
 77   }
 78 
 79   return {
 80     hits,
 81     scannedFiles,
 82     scope,
 83   };
 84 }
